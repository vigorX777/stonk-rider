# 韭菜骑士项目全面复盘

> 复盘范围：从最初创意、设计与开发，到数据、玩法、测试、品牌、GitHub、Cloudflare Pages + D1 部署，以及 1.3.5 的连续迭代。
>
> 当前版本：1.3.5  
> 公网地址：https://leek-knight.pages.dev  
> 代码仓库：https://github.com/vigorX777/leek-knight

## 1. 总体结论

韭菜骑士在约两天内完成了从想法到公网产品的闭环：

- 把 20 支 A 股近一年真实日线转换成摩托车物理赛道。
- 建立可测试的赛道、收益、控制、Combo 和 BGM 纯函数。
- 完成严格金币拾取、真实行情结算、摔车复位、跳跃、Combo、反馈动画和程序化音频。
- 加入 Cloudflare Pages Functions + D1 排行榜。
- 完成公开 GitHub 仓库、线上试玩、品牌改名、README 配图和作者信息。
- 当前生产站可访问，排行榜已有真实玩家数据。

这是一次成功的“快速产品化”实践。项目不是停留在原型，而是完成了可玩、可分享、可验证、可部署、可继续运营的最小产品。

同时，开发过程也暴露了三类核心问题：

1. **手感验收滞后**：动力、姿态、坡度、跳跃、金币碰撞和音量主要依赖上线后的人工试玩反复校准。
2. **文档与实现漂移**：部分设计文档仍记录旧跳跃条件、旧力度单位和不准确的限流描述。
3. **公网竞争完整性不足**：排行榜验证了输入格式和金额自洽，但不能证明成绩确实来自一次合法游戏过程。

下一阶段不应继续无序堆功能。应先补齐“成绩可信、自动化验收、代码拆分、文档真相源”四个基础能力，再发展每日挑战、分享传播和赛季玩法。

### 2.1 立项阶段

最初目标非常清晰：把 A 股 K 线直接变成摩托车爬坡赛道，并把上涨、下跌映射成正负收益金币。

初始需求已经包含了产品最重要的差异化：

- 真实 A 股近一年走势，而不是随机生成地图。
- 摩托物理操作，而不是单纯看图或模拟交易。
- 初始金额和收益结算，强化市场波动的戏剧性。
- 音效、平衡、加速、刹车、重启等完整游戏操作。

这个立项的优点是“核心隐喻一句话就能说清楚”：骑着摩托跑完一支股票的年度 K 线。

### 2.2 1.0.0：最小可玩版本

第一版完成了：

- Vite + TypeScript + Phaser Matter Physics 工程。
- 20 支 A 股固定行情快照。
- K 线赛道、摩托车刚体、平衡、翻车和通关。
- 区间收益结算、HUD、结果统计和程序化音效。
- 运行时不依赖行情接口或外部 CDN。

这一阶段最重要的正确决策是：**行情数据在构建期抓取，浏览器运行时只读取固定 JSON 快照**。它让游戏具备可复现性，也避免行情 API、跨域、限流和数据漂移影响游戏体验。

### 2.3 1.0.1：数据真实性与基础控制修复

首次试玩很快暴露两个根本问题：

- 长按前进不能持续提供动力。
- 不同股票走势出现重复或无法体现完整年度差异。

对应改造包括：

- 改为持续后轮驱动。
- 股票卡片使用各自完整年度收盘序列。
- 新增完整年度 K 线总览和当前位置游标。
- 加入数据去重回归测试。

这说明“数据可信”不是辅助能力，而是产品核心。如果赛道不是对应股票的真实独立走势，整个产品概念就失效。

### 2.4 1.1.x：物理手感重构

动力不足、姿态键无效、坡度太平、陡坡卡车、音效解锁失败等问题连续出现。

1.1.0 的关键转折是先调研成熟物理驾驶游戏和 Matter 官方实现，再新增 `bikeControl.ts` 纯函数控制器：

- 后轮角速度和切向牵引力共享同一扭矩模型。
- 移除互相竞争的车架速度修正。
- 区分低速扭矩、坡度补偿、最高速度和防失速。
- 将姿态输入、角速度目标、载荷转移纳入统一控制逻辑。
- 用 HUD 显示抓地、腾空、油门和驱动模式。

随后继续通过试玩修正：

- 提升动力和坡度振幅。
- 取消双重平滑，保留 K 线局部高低点。
- 增加起步助跑平台。
- 摔车后原地复位，不中断整局。
- 修复左右姿态控制和 Web Audio 解锁。

这阶段证明：**游戏手感不能靠一个推力常量解决，必须把动力来源、抓地、质量、约束、空中控制和反馈作为一个系统设计。**

### 2.5 1.1.4-1.2.0：玩法与刺激反馈

在基础驾驶可用后，目标从“能玩”转向“有刺激感”：

- 金币从越过坐标自动结算改为严格物理拾取。
- 金币间隔由 5 个交易日调整为 8 个交易日。
- 增加高位跳跃金币。
- 涨金币使用红色 `+`，跌金币使用绿色 `-`。
- 增加双 Combo、hit-stop、震动、闪色、粒子和金额滚动。
- 未拾取金币保留，允许倒车重新获取。
- 加入真实跳跃和落地牵引渐变。

这一阶段采用了设计文档、实现计划和独立任务拆分。对音频、DOM UI、粒子等低冲突模块有效，但核心物理仍集中在 `RideScene.ts`，不适合过度并行。

### 2.6 1.3.0：排行榜与公网产品化

排行榜让项目从单机原型变成具有社交比较的在线产品：

- Cloudflare Pages 托管静态前端。
- Pages Functions 提供同域 API。
- D1 保存成绩。
- 同一匿名玩家同一股票只保留最高分。
- 支持单股榜、全球榜、Top 3、个人最佳和百分位。
- API 校验股票、昵称、UUID、金额、进度和收益率格式。

部署阶段还完成了：

- 创建真实 D1 并执行 schema。
- 验证公网 GET/POST 和非法提交拒绝。
- 创建公开 GitHub 仓库。
- README 增加试玩链接、仓库地址、作者信息和项目插图。

### 2.7 1.3.1-1.3.5：品牌、跳跃和 BGM

后续完成了三类产品修整：

- 品牌从 Stonk Rider 迁移为「韭菜骑士 / Leek Knight」。
- GitHub、Cloudflare Pages、D1、包名、存储 key 和内部类名同步改名。
- 跳跃改为任一车轮着地即可触发，并增加 140ms 输入缓冲。
- 新增程序化 8-bit BGM，处理浏览器自动播放策略，并多次调整可听音量。

这部分提升了品牌辨识度和完成感，但也再次说明：**音频是否“运行”不等于用户是否“听得见”，自动化状态验证不能替代真实听感验收。**

## 3. 做得好的地方

### 3.1 产品概念足够独特

“真实 A 股 K 线就是摩托赛道”同时具备：

- 一句话传播能力。
- 可视化记忆点。
- 市场情绪与物理游戏的天然冲突感。
- 每支股票自动形成不同关卡的内容扩展能力。

项目不需要大量解释，用户看到画面就能理解核心玩法。

### 3.2 技术选型与规模匹配

当前架构基本合理：

- Phaser + Matter 处理高频物理和 Canvas 渲染。
- DOM 处理表单、HUD、排行榜和可访问文本。
- Vite + TypeScript 保持工程轻量。
- 没有为单页面小游戏引入 React 或复杂状态框架。
- Cloudflare Pages + Functions + D1 足以支撑当前规模。

这体现了良好的 YAGNI：先解决真实产品问题，没有为假想规模提前建设复杂后端。

### 3.3 关键业务规则保持纯函数

以下逻辑被分离并获得单元测试：

- `track.ts`：行情到赛道与结算点转换。
- `balance.ts`：收益、Combo 和格式化。
- `bikeControl.ts`：动力、制动、倒车、姿态和跳跃输出。
- `chart.ts`：缩略图几何。
- `bgm.ts`：程序化 BGM 节拍。

这使最容易出错的规则可以脱离 Phaser 场景验证，是项目最值得保留的工程习惯。

### 3.4 数据和物理坐标边界清楚

项目明确规定：

- 赛道可以为了可玩性调整坡度。
- 收益必须来自原始行情区间。
- 物理坐标不得反向参与收益计算。

这是正确的领域边界。它允许“游戏化地形”和“真实行情来源”同时存在，不会因坡度修正污染收益逻辑。

### 3.5 用户试玩反馈推动了真实质量

多轮反馈直接发现了自动测试不容易捕捉的问题：

- 长按油门不持续。
- 左右姿态控制被物理约束抵消。
- K 线过度平滑。
- 缩略图与赛道感知不一致。
- 金币碰撞过宽。
- 跳跃输入窗口不跟手。
- BGM 虽处于 running 状态但实际听不见。

这说明持续试玩是游戏项目不可替代的质量环节。

### 3.6 交付闭环完整

项目不是只完成代码：

- 有项目规则、架构、数据、研究、ADR、玩法和 changelog。
- 有 9 个测试文件、73 个测试用例。
- 有公网服务、数据库、公开仓库和 README。
- 有品牌、作者、插图和试玩链接。

从作品完整度看，已经超过普通临时 demo。

## 4. 可以优化的地方

### 4.1 Git 建仓太晚

最早可见提交已经是 1.3.0，一次性包含 68 个文件和约 8800 行新增内容。1.0.0 到 1.2.0 的过程只能从 changelog 和任务记录还原。

影响：

- 无法精确比较早期每次物理调参。
- 无法单独回滚某个玩法阶段。
- 代码审查和问题归因成本增大。
- 复盘依赖人工叙事，而不是提交证据。

下次应在创建工程骨架时立即 `git init`，至少按“基础工程、数据管线、赛道、车辆、收益、UI、部署”拆分提交。

### 4.2 计划文档过长，但项目真相仍发生漂移

两个实现计划合计超过 2400 行，但当前仍存在：

- `GAME_DESIGN.md` 还写“后轮着地才能跳”。
- Combo 设计概述仍残留旧后轮规则。
- 设计文档写跳跃力 `0.055~0.10`，实际控制函数返回 `8~13` 的速度增量语义。
- 排行榜设计写“Pages Functions 自带 10 req/s 限流”，实现并没有对应限流代码或明确配置。
- 历史目录名、占位 database id 等仍留在设计文档。

根因不是文档不够多，而是缺少明确真相源和发布前一致性检查。

建议只保留四层文档：

1. `README.md`：用户入口。
2. `GAME_DESIGN.md` / `ARCHITECTURE.md`：当前事实。
3. ADR：重要取舍。
4. 历史 spec/plan：仅作过程归档，并明确标记“非当前真相”。

### 4.3 核心文件开始过大

当前主要文件规模：

- `RideScene.ts`：755 行。
- `main.ts`：645 行。
- `functions/api/leaderboard.js`：351 行。
- `audio.ts`：280 行。

问题不是单纯行数，而是职责混合：

- `RideScene.ts` 同时负责输入、车辆物理、碰撞、金币、Combo、粒子、相机、摔车和结算。
- `main.ts` 同时负责页面模板、状态、HUD、排行榜、动画、音频生命周期和事件绑定。
- 排行榜 API 同时负责校验、SQL、排名、序列化和 CORS。

下一次较大功能开发前应先拆分，而不是继续往这些文件追加分支。

### 4.4 手感调试缺少标准化工具

动力、坡度、姿态、跳跃和音频经历了大量“试玩反馈 -> 调常量 -> 再部署”的循环。

缺少的工具包括：

- 固定的测试股票/测试路段集合。
- 输入录制与回放。
- 车辆关键参数实时调试面板。
- 单轮着地、陡坡起步、落地、急跌等场景脚本。
- 物理数据采样和回归阈值。
- 音频总线电平或波形分析。

下次遇到类似物理游戏，应先建立“手感实验场”，再调正式关卡。

### 4.5 浏览器验证是临时脚本，不是仓库能力

目前已经多次使用 Playwright 验证：

- 首页可加载。
- 可进入选股和游戏。
- Canvas 正常生成。
- 音频状态正常。
- 控制台无应用错误。

但这些脚本没有进入仓库，所以每次都需要重新编写，无法在 CI 中回归。

建议增加正式 E2E：

- 首页 -> 车库 -> 开局。
- 音频首次交互解锁。
- 跳跃输入与 HUD 状态。
- 通关后排行榜提交。
- API 失败时游戏主体仍可运行。

### 4.6 排行榜成绩可以伪造

API 当前验证了：

- 股票代码和名称合法。
- 昵称和 UUID 合法。
- `progress === 1`。
- `final = initial × (1 + return_rate)`。
- 收益率在允许范围内。

但客户端仍可直接构造任意合法范围内的收益率。服务器没有重放金币拾取、Combo、股票快照版本和游戏规则。

这意味着排行榜目前适合作为轻量社交功能，不适合作为严肃竞争系统。

可选修复路径：

1. **轻量方案**：服务端根据股票、初始金额、已拾取结算点和 Combo 序列重算结果。
2. **中等方案**：开局由服务端签发 run token，提交压缩事件日志，服务端校验时间和结算序列。
3. **严格方案**：确定性输入回放或服务端模拟，但复杂度明显更高。

当前规模建议采用第二种方案，不需要完整服务端物理，只验证与收益相关的事件。

### 4.7 “真实收益”表述需要更准确

结算使用真实历史行情区间，但 Combo 会放大涨跌影响，因此最终余额并不等同于真实持有该股票的历史收益。

更准确的对外表述是：

> 基于真实历史涨跌的游戏化收益结算。

这能保留产品亮点，同时避免把游戏结果误解成真实投资回测。

### 4.8 部署流程仍依赖人工

当前发布链路已跑通，但仍以手工命令为主：

- 手工测试、构建。
- 手工 `wrangler pages deploy`。
- 手工公网 curl/Playwright 验证。
- 手工更新共享任务状态。

风险包括漏测、部署错误目录、生产配置漂移和发布记录不完整。

建议建立 GitHub Actions：

- PR：typecheck + test + build + E2E。
- main：自动部署 Cloudflare Pages。
- 发布：生成版本标签和 release notes。
- 部署后：健康检查首页和排行榜 GET。

### 4.9 资源和基础设施需要清理

项目改名时创建了新的 `leek-knight-db` 和 Pages 项目。旧的 `stonk-rider-db` 仍可能保留在 Cloudflare 账户中。

此外，仓库还有 Vite 默认 SVG、旧 hero 图片等未引用资源，应通过依赖扫描确认后清理。

### 4.10 当前产品验证样本仍小

生产排行榜目前只有少量玩家数据。可以确认技术链路可用，但还不足以判断：

- 玩家是否愿意完成整条年度赛道。
- 哪些股票最受欢迎。
- 平均在哪个阶段退出或摔车。
- Combo、跳跃金币和排行榜是否真正提升留存。
- BGM 和强刺激反馈是否会造成疲劳。

下一阶段需要事件分析，而不是继续凭感觉增加功能。

## 5. 可复用、可回流的经验

### 5.1 目标信号：先保护产品的核心隐喻

当产品卖点是“真实数据生成游戏内容”时，数据真实性和数据差异必须先于视觉润色验收。

可复用规则：

> 任何由真实数据驱动的创意产品，第一批回归测试必须覆盖数据来源、时间范围、唯一性和展示映射，否则后续玩法优化没有可信基础。

### 5.2 判断信号：物理手感是系统问题

动力不足不一定只需要增加推力，姿态无效也不一定只需要增加扭矩。质量、约束、抓地、角速度、坡面、输入时机和反馈会互相影响。

可复用规则：

> 调整物理手感前，先统一动力来源并移除互相竞争的速度/力修正，再通过可测试控制器调整参数。

### 5.3 动作信号：纯函数隔离高风险规则

赛道、收益、车辆控制、图表和音乐节拍都适合先写成纯函数，场景层只负责采集状态和应用结果。

可复用规则：

> 高频实时系统中，凡是能描述为“输入状态 -> 输出参数”的规则，都优先从引擎生命周期中抽离成纯函数并加回归测试。

### 5.4 动作信号：人工试玩必须变成测试矩阵

试玩发现的问题非常有价值，但不能每次从零开始。

建议固定矩阵：

| 场景 | 验收点 |
|---|---|
| 平地起步 | 长按动力持续、松开滑行 |
| 陡坡低速 | 不失速、不瞬移 |
| 前轮单独着地 | 可跳跃、可防卡坡 |
| 腾空 | 左右姿态持续有效 |
| 落地 | 无异常弹射、牵引渐入 |
| 急跌 | 镜头预判、制动可控 |
| 摔车 | 原地复位、状态保留 |
| 金币 | 碰撞严格、正负符号正确 |
| 音频 | 实际可听、静音有效、首次交互可恢复 |

### 5.5 边界信号：运行状态不等于用户感知

`AudioContext=running`、`bgmRunning=true` 只能证明程序执行，不能证明用户听得见。Canvas 存在也不能证明手感正常。

可复用规则：

> 对声音、动画、物理和视觉反馈，自动化负责验证“系统运行”，人工验收负责验证“用户感知”，两者不能互相替代。

### 5.6 动作信号：部署后验证要清理测试数据

本项目公网 API 验证曾写入测试成绩，随后主动删除。

可复用规则：

> 生产写入型验证必须使用可识别测试标记，并在同一流程内自动清理；不能把清理留给记忆或人工。

### 5.7 判断信号：计划粒度应匹配风险，不匹配文件数量

排行榜适合按 API、D1、客户端和 UI 拆分；物理手感不适合把多个代理同时写入同一个场景文件。

可复用规则：

> 并行任务必须同时满足“文件边界独立”和“行为状态独立”；共享实时状态或同一核心文件的任务应串行实现和实玩验收。

### 5.8 边界信号：公共排行榜不能信任客户端结果

格式校验和金额自洽不等于成绩可信。

可复用规则：

> 任何公开竞争排行，只要结果影响名次，服务端就必须掌握可重算的规则输入或可信执行证据，不能直接接受客户端最终分数。

## 6. 建议回流候选

以下经验具有跨项目复用价值，适合进入开发经验候选区，而不只是留在韭菜骑士项目文档中：

1. **真实数据驱动产品的首轮验证**：先验证时间范围、唯一性、映射关系和失败回退，再做玩法与视觉。
2. **物理游戏的双层验收**：纯函数/状态自动化验证系统，固定场景试玩验证手感。
3. **浏览器音频的真实验收边界**：AudioContext 状态只能验证执行，不能验证响度与听感。
4. **公开排行榜的信任边界**：客户端结果只能作为声明，服务器必须重算关键业务结果。
5. **生产写入验证自动清理**：测试标识、写入、查询和删除应为同一个可重复脚本。

## 7. 后续发展方向

### P0：先补可信度与可维护性

1. **排行榜可信提交**
   - 服务端签发 run token。
   - 提交股票快照版本、拾取金币序列、Combo 序列和用时。
   - 服务端重算最终金额。

2. **拆分核心文件**
   - `RideScene` 拆为车辆、金币结算、Combo/反馈、相机与复位模块。
   - `main.ts` 拆为 landing/setup/game/result/leaderboard UI 控制器。
   - API 拆为 validation、repository、ranking 和 response。

3. **建立正式 E2E 与发布流水线**
   - Playwright 流程进入仓库。
   - GitHub Actions 自动测试、构建和部署。
   - 生产健康检查自动化。

4. **文档对齐**
   - 修正跳跃条件和力度单位。
   - 标记历史计划为 archive/reference。
   - 让 `GAME_DESIGN.md` 和 `ARCHITECTURE.md` 成为当前真相。

### P1：提高传播和重复游玩

1. **每日挑战**
   - 每天固定一支股票、初始金额和规则版本。
   - 所有人跑同一赛道，排行榜更公平。

2. **结果分享卡片**
   - 股票、收益率、全球排名、最强 Combo、摔车次数。
   - 自动生成适合朋友圈和社交平台的图片。

3. **短赛道模式**
   - 年度赛道对新玩家可能过长。
   - 增加 1 个月、1 季度或“最刺激 30 天”模式。

4. **新手引导**
   - 第一次进入时用 20-30 秒解释加速、姿态、跳跃和金币。
   - 用安全训练坡代替文字说明。

5. **幽灵回放**
   - 显示自己最佳记录或排行榜前列玩家的路线。
   - 增加学习与挑战动机。

### P2：增强内容与运营

1. **赛季榜和股票主题周**。
2. **挑战成就**：零摔车、全金币、极限跳跃、负收益生存。
3. **可分享的股票赛道链接**：URL 直接指定股票和模式。
4. **行情快照版本化**：记录抓取日期、数据哈希和赛季规则版本。
5. **基础产品分析**：开局、选股、完成率、摔车位置、金币遗漏和排行榜提交率。

### P3：谨慎评估的方向

1. **移动端触控**：传播价值高，但需要重新设计操作和横屏布局。
2. **多人实时竞速**：吸引力强，但复杂度和反作弊成本高。
3. **UGC 股票池**：需要数据授权、刷新机制和赛道质量控制。
4. **车辆养成**：可能破坏不同股票赛道之间的公平比较，应放在独立娱乐模式。

## 8. 推荐的下一项目工作流

### 阶段 A：立项

- 用一句话定义核心体验。
- 明确非目标和不可破坏边界。
- 创建 Git 仓库并提交工程骨架。
- 写 1 页产品 brief，不先写超长计划。

### 阶段 B：风险原型

- 先验证最大风险：数据真实性、物理手感或浏览器能力。
- 建立 3-5 个固定测试场景。
- 把核心规则拆成纯函数并先加测试。

### 阶段 C：可玩版本

- 完成最短闭环：选择 -> 操作 -> 反馈 -> 结束。
- 每个阶段独立提交。
- 使用参数面板调手感，不直接反复改常量。

### 阶段 D：体验增强

- 先定义体验目标和可观察指标。
- 视觉、音效和动画分别设置主次层级。
- 做固定试玩矩阵并记录结果。

### 阶段 E：联网与部署

- 先定义客户端与服务端信任边界。
- 为生产验证设计可自动清理的数据。
- 部署前执行 test、build、E2E 和文档一致性检查。
- 先发 preview，再提升到 production。

### 阶段 F：发布后

- 收集行为数据，而不是只收主观反馈。
- 合并同类微调，减少每个常量都单独发版。
- 每个版本记录“为什么改、如何验证、是否达到目标”。

## 9. 下一步建议

如果继续投入，本项目最合理的下一阶段不是增加更多特效，而是完成以下三个闭环：

1. **公平闭环**：排行榜成绩可验证。
2. **质量闭环**：固定场景 E2E + 手感测试矩阵 + 自动发布。
3. **传播闭环**：每日挑战 + 分享卡片 + 短赛道。

完成这三项后，韭菜骑士才会从“完成度很高的创意小游戏”进一步变成“可持续传播和运营的小产品”。
